# Переменные окружения

Пример находится в `.env.example`. Настоящий `.env` содержит секреты и не должен попадать в git.

## APP_ENV

- Назначение: режим работы приложения.
- Пример: `production`.
- Можно менять после запуска: да, затем перезапустить контейнеры.
- Секрет: нет.

## APP_TIMEZONE

- Назначение: часовой пояс сервера для Django и проверок расписаний.
- Пример: `Europe/Moscow`.
- Можно менять после запуска: да, но лучше выбрать один раз до начала эксплуатации.
- Секрет: нет.

## POSTGRES_DB

- Назначение: имя базы данных PostgreSQL.
- Пример: `machine_control`.
- Можно менять после запуска: только вместе с переносом/созданием новой БД.
- Секрет: нет.

## POSTGRES_USER

- Назначение: пользователь PostgreSQL.
- Пример: `machine_control`.
- Можно менять после запуска: только вместе с настройкой роли в PostgreSQL.
- Секрет: частично, лучше не публиковать.

## POSTGRES_PASSWORD

- Назначение: пароль пользователя PostgreSQL.
- Пример: `change_me_strong_password`.
- Можно менять после запуска: да, но нужно синхронно изменить пароль в БД и `.env`.
- Секрет: да.

## POSTGRES_HOST

- Назначение: имя контейнера PostgreSQL внутри Docker-сети.
- Пример: `postgres`.
- Можно менять после запуска: обычно нет, если не меняется compose-сервис.
- Секрет: нет.

## POSTGRES_PORT

- Назначение: порт PostgreSQL внутри Docker-сети.
- Пример: `5432`.
- Можно менять после запуска: обычно нет.
- Секрет: нет.

## DJANGO_SECRET_KEY

- Назначение: секретный ключ Django для подписи cookies и служебных токенов.
- Пример: длинная случайная строка.
- Можно менять после запуска: нежелательно, активные сессии пользователей могут стать недействительными.
- Секрет: да.

## DJANGO_DEBUG

- Назначение: включает debug-режим Django.
- Пример: `false`.
- Можно менять после запуска: да, затем перезапустить `control-web`.
- Секрет: нет.
- Production: должно быть `false`.

## DJANGO_ALLOWED_HOSTS

- Назначение: список hostnames/IP, с которых Django принимает HTTP-запросы.
- Пример: `127.0.0.1,localhost,indctrl.example.local`.
- Можно менять после запуска: да, затем перезапустить `control-web`.
- Секрет: нет.

## DJANGO_CSRF_TRUSTED_ORIGINS

- Назначение: доверенные origin для Django CSRF.
- Пример: `http://indctrl.example.local`.
- Можно менять после запуска: да, затем перезапустить `control-web`.
- Секрет: нет.

## AUTH_SERVICE_PORT

- Назначение: справочная переменная порта auth-service.
- Пример: `8000`.
- Можно менять после запуска: в текущем compose не используется для публикации наружу.
- Секрет: нет.

## EVENT_SERVICE_PORT

- Назначение: справочная переменная порта event-service.
- Пример: `8000`.
- Можно менять после запуска: в текущем compose не используется для публикации наружу.
- Секрет: нет.

## CONTROL_WEB_PORT

- Назначение: справочная переменная порта control-web.
- Пример: `8000`.
- Можно менять после запуска: в текущем compose не используется для публикации наружу.
- Секрет: нет.

## SESSION_TTL_MINUTES

- Назначение: время жизни сессии работника после login на ESP32.
- Пример: `720`.
- Можно менять после запуска: да, влияет на новые сессии после перезапуска сервисов.
- Секрет: нет.

## HEARTBEAT_MAX_AGE_SECONDS

- Назначение: максимальный возраст heartbeat, при котором dashboard считает связь активной.
- Пример: `180`.
- Можно менять после запуска: да, затем перезапустить сервисы.
- Секрет: нет.

## BACKUP_RETENTION_DAYS

- Назначение: сколько дней хранить backup-файлы PostgreSQL.
- Пример: `30`.
- Можно менять после запуска: да, применяется при следующем запуске backup-скрипта.
- Секрет: нет.

## BACKUP_DIR

- Назначение: каталог для backup-файлов PostgreSQL.
- Пример: `./backups/postgres`.
- Можно менять после запуска: да, убедитесь, что у пользователя есть права записи.
- Секрет: нет, но сам каталог backup надо защищать правами доступа.
