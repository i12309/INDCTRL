# Архитектура INDCTRL

INDCTRL - система контроля работы станков на одном Linux-сервере. Проект
состоит из нескольких Docker-контейнеров, которые работают в общей внутренней
сети и используют одну PostgreSQL-БД.

## Общая схема сервисов

- `nginx` - единая HTTP-точка входа. Наружу публикуется только порт `80`.
- `control-web` - Django-сервис для `/admin/`, `/dashboard/`, `/reports/`,
  CSV/XLSX-экспорта и миграций БД.
- `auth-service` - FastAPI-сервис авторизации работников на ESP32.
- `event-service` - FastAPI-сервис приема событий о произведенных деталях.
- `postgres` - общая PostgreSQL-БД.

Потоки данных:

```text
ESP32 -> nginx -> auth-service  -> PostgreSQL
ESP32 -> nginx -> event-service -> PostgreSQL
браузер -> nginx -> control-web -> PostgreSQL
```

## Почему PostgreSQL

PostgreSQL выбран как основная БД, потому что системе нужны транзакции,
уникальные ограничения, индексы для отчетов и корректная конкурентная запись.
Важные правила проекта защищены на уровне БД:

- один активный `Work` на один станок;
- уникальность детали внутри смены по `user_id`, `machine_id`, `work_id`,
  `detail_number`;
- уникальность MAC-адреса ESP32-устройства;
- уникальность базового разрешения пользователя на станок.

SQLite для production-схемы не подходит: несколько сервисов пишут в одну БД,
а отчеты и прием событий могут работать одновременно.

## Почему Docker Compose

Production-запуск рассчитан на один Linux-сервер. Docker Compose дает:

- повторяемый запуск всех сервисов одной командой;
- изоляцию Python-зависимостей каждого сервиса;
- общую внутреннюю сеть без публикации PostgreSQL наружу;
- volume для данных PostgreSQL и static files;
- простые команды обслуживания: миграции, логи, backup и restart.

## Почему FastAPI для auth-service и event-service

`auth-service` и `event-service` обслуживают ESP32 и должны быстро принимать
короткие JSON-запросы. FastAPI хорошо подходит для таких HTTP API: схемы Pydantic
фиксируют контракт, обработчики остаются небольшими, а бизнес-логика вынесена в
сервисные классы и репозитории.

FastAPI-сервисы не запускают миграции и не используют Django ORM. Они обращаются
к таблицам напрямую через общий пакет `control_common` и SQL-запросы.

## Почему admin, dashboard и reports в control-web

Админка, dashboard и отчеты объединены в одном Django-сервисе `control-web`,
потому что они используют один набор моделей, авторизацию пользователей,
шаблоны, формы и ORM. Внутри Django ответственность разделена по приложениям:

- `accounts` - пользователи и роли;
- `machines` - станки и ESP32-устройства;
- `schedules` - разрешения и расписания;
- `production` - смены, сессии, детали, ошибки событий;
- `dashboard` - текущие работники на станках;
- `reports` - отчеты и экспорт.

Django-модели и Django migrations являются основным источником структуры БД.
Все изменения схемы должны проходить через `control-web`.

## Взаимодействие через общую БД

Сервисы не вызывают друг друга напрямую.

`auth-service`:

- читает `Device`, `Machine`, `User`, `Role`, разрешения и расписания;
- при успешном login создает `Work` и `AuthSession`;
- при heartbeat обновляет `Work.last_seen_at`;
- при logout закрывает `AuthSession` и завершает `Work`.

`event-service`:

- читает активную `AuthSession`, связанную активную `Work`, `DetailType` и
  `DetailState`;
- сохраняет `Detail`;
- сохраняет `InvalidEvent`, если входящее событие нельзя принять.

`control-web`:

- создает и изменяет справочники, пользователей, станки, устройства, разрешения
  и расписания через admin;
- читает рабочие смены и детали для dashboard и отчетов;
- выполняет миграции.

## Границы ответственности

`auth-service` отвечает только за вход работника на станок, список доступных
работников, heartbeat и logout. Он не принимает детали и не строит отчеты.

`event-service` отвечает только за прием событий деталей. Он не проверяет пароль
работника и не создает смены.

`control-web` отвечает за административное управление, отчеты, dashboard и схему
БД. Он не является API для ESP32.

`nginx` маршрутизирует HTTP-запросы и отдает static files. Бизнес-логику он не
содержит.

`postgres` хранит все данные проекта. Прямой доступ к нему снаружи сервера не
нужен для штатной работы.
