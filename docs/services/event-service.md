# event-service API

`event-service` - FastAPI-сервис приема событий о произведенных деталях от ESP32.
Он проверяет активную сессию, сохраняет новую деталь идемпотентно и пишет
диагностические ошибки в `InvalidEvent`.

## GET /health

Назначение: проверить, что контейнер отвечает.

Пример запроса:

```bash
curl http://localhost/health-events/
```

Успешный ответ:

```json
{"status": "ok", "service": "event-service"}
```

Пример ошибки: если контейнер не запущен, Nginx вернет `502 Bad Gateway`.

Что меняется в БД: ничего.

## POST /api/events/detail

Назначение: принять событие о произведенной детали.

Рекомендуемый JSON:

```json
{
  "sessionID": "11111111-1111-1111-1111-111111111111",
  "time": "2026-05-17 14:30:00",
  "detail": {
    "number": 1,
    "type": 1,
    "state": 1
  }
}
```

Допустимо дополнительно прислать `userID`, `machineID`, `workID`, но они будут
только сверены с активной сессией:

```json
{
  "sessionID": "11111111-1111-1111-1111-111111111111",
  "userID": 10,
  "machineID": 1,
  "workID": 100,
  "time": "2026-05-17 14:30:00",
  "detail": {
    "number": 1,
    "type": 1,
    "state": 1
  }
}
```

Успешный ответ для новой детали:

```json
{"success": true, "status": "saved"}
```

Успешный ответ для повтора той же детали:

```json
{"success": true, "status": "duplicate"}
```

Пример ошибки:

```json
{"success": false, "error": "Активная сессия не найдена"}
```

Что меняется в БД:

- при новой детали создается `details`;
- при новой детали и при дубле обновляется `works.last_seen_at`;
- при ошибках входного JSON, структуры, сессии, типа детали, состояния детали или
  несовпадении optional ID создается `invalid_events`.

## Роль sessionID

`sessionID` - UUID из `auth_sessions`, который ESP32 получает после login.
Именно по нему `event-service` определяет пользователя, станок и смену.

Значения `userID`, `machineID`, `workID` из запроса не являются источником
истины. Если они переданы и не совпадают с `AuthSession`, событие отклоняется и
попадает в `InvalidEvent`.

## Идемпотентность

Деталь уникальна по:

```text
user_id, machine_id, work_id, detail_number
```

`event-service` использует `INSERT ... ON CONFLICT DO NOTHING`. Поэтому ESP32
может безопасно повторить отправку после сетевого сбоя:

- если запись создана впервые, сервис возвращает `saved`;
- если такая деталь уже есть, сервис возвращает `duplicate`.

## Когда создается InvalidEvent

`InvalidEvent` создается, когда сервис не может принять событие как корректную
деталь:

- body не является JSON;
- структура не соответствует схеме;
- `sessionID` неизвестен, отключен или истек;
- связанная смена уже не активна;
- `time` нельзя разобрать;
- тип детали или состояние детали не найдены либо отключены;
- `userID`, `machineID` или `workID` не совпадают с активной сессией.
