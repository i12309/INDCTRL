# Задача 07. Реализовать отчеты и dashboard в Django `control-web`

## Цель

Сделать первые отчетные страницы в Django: dashboard текущей работы станков и отчет по произведенным деталям.

## Общие обязательные требования

- Проект называется `INDCTRL`.
- Язык: Python 3.12+.
- База данных: PostgreSQL.
- Запуск в production: Docker Compose на одном Linux-сервере.
- Сервисы:
  - `auth-service` — FastAPI-сервис авторизации работников на ESP32;
  - `event-service` — FastAPI-сервис приема событий о произведенных деталях;
  - `control-web` — Django-сервис, который содержит админку, dashboard и отчеты;
  - `postgres` — общая PostgreSQL-БД;
  - `nginx` — единая точка входа для HTTP.
- Админка и отчеты должны быть в одном Django-сервисе `control-web`. Внутри Django их надо разделить по приложениям/модулям.
- Django-модели и Django migrations являются основным источником структуры БД.
- FastAPI-сервисы работают с этой же БД напрямую через общий пакет и SQL-запросы, но не запускают миграции.
- Весь код должен быть понятен программисту, который плохо знает Python, FastAPI и Django.
- Все публичные классы, Django-модели, Pydantic-схемы, функции, методы и сервисные классы должны иметь докстринги или комментарии на русском языке.
- Сложная бизнес-логика должна быть пояснена комментариями на русском языке: проверки прав, расписаний, активных смен, идемпотентности, обработки дублей и ошибок.
- Комментарии должны объяснять смысл и причину решения, а не дублировать очевидное действие кода.
- В каждом сервисе должна быть собственная документация `README.md`: назначение, переменные окружения, запуск в Docker, основные endpoint'ы, диагностика и просмотр логов.

## Почему отчеты делаются на Django

Отчеты остаются внутри `control-web`, потому что Django уже имеет ORM, шаблоны, формы, авторизацию и доступ к тем же моделям, что и админка.

Структурно отчеты должны быть отдельным Django-приложением:

```text
apps/reports/
```

Dashboard — отдельным приложением:

```text
apps/dashboard/
```

## Доступы

### Dashboard

Доступ:

```text
admin
director
manager
```

### Reports

Доступ:

```text
admin
director
manager
```

`worker` не должен иметь доступ к отчетам и dashboard.

## Dashboard: кто сейчас работает

URL:

```http
GET /dashboard/current-workers/
```

Показать таблицу активных смен:

```text
Станок
Работник
Начало смены
Последний heartbeat
Статус связи
```

Логика статуса связи:

```text
если last_seen_at свежий — связь активна;
если last_seen_at старше HEARTBEAT_MAX_AGE_SECONDS — связь устарела;
если last_seen_at пустой — heartbeat еще не приходил.
```

Данные брать из `Work`, где `status='active'`.

## CSV-экспорт

URL:

```http
GET /reports/details/export/csv/
```

Экспорт должен учитывать те же фильтры, что и HTML-страница.

## Excel-экспорт

URL:

```http
GET /reports/details/export/xlsx/
```

Использовать `openpyxl`.

Excel-файл должен содержать:

```text
лист Details;
заголовки колонок на русском;
строки отчета;
итоговые значения.
```

PDF в первой версии не делать.

## Структура кода

```text
forms.py       формы фильтров;
services.py    запросы и подготовка данных отчета;
views.py       Django views;
templates/     HTML-шаблоны.
```

Не размещать всю логику отчета прямо во `views.py`.

## Производительность

Для запросов использовать:

```text
select_related для user, machine, work, detail_type, detail_state;
фильтрацию на уровне БД;
пагинацию;
```

Не загружать в память все детали за годы для HTML-страницы.

Для Excel/CSV можно ограничить максимальное количество строк на первой версии, например 100 000, и показать понятную ошибку, если отчет слишком большой.

## Nginx

Маршруты:

```text
/reports/   -> control-web:8000
/dashboard/ -> control-web:8000
```

## Критерии приемки

- `/dashboard/current-workers/` показывает активные смены.
- `/reports/details/` показывает таблицу деталей с фильтрами.
- Работают фильтры по датам, станку, работнику, смене, типу и состоянию.
- Работает CSV-экспорт.
- Работает XLSX-экспорт.
- Роль `worker` не имеет доступа к отчетам.
- Код отчетов разделен на forms/services/views.
