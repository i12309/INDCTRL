# Задача 05. Реализовать `event-service` на FastAPI

## Цель

Создать FastAPI-сервис, который принимает события о произведенных деталях от ESP32, проверяет активную сессию, сохраняет детали в PostgreSQL и обрабатывает дубли идемпотентно.

## Общие обязательные требования

- Проект называется `INDCTRL`.
- Язык: Python 3.12+.
- База данных: PostgreSQL.
- Запуск в production: Docker Compose на одном Linux-сервере.
- Сервисы:
  - `auth-service` — FastAPI-сервис авторизации работников на ESP32;
  - `event-service` — FastAPI-сервис приема событий о произведенных деталях;
  - `control-web` — Django-сервис, который содержит админку, dashboard и отчеты;
  - `postgres` — общая PostgreSQL-БД;
  - `nginx` — единая точка входа для HTTP.
- Админка и отчеты должны быть в одном Django-сервисе `control-web`. Внутри Django их надо разделить по приложениям/модулям.
- Django-модели и Django migrations являются основным источником структуры БД.
- FastAPI-сервисы работают с этой же БД напрямую через общий пакет и SQL-запросы, но не запускают миграции.
- Весь код должен быть понятен программисту, который плохо знает Python, FastAPI и Django.
- Все публичные классы, Django-модели, Pydantic-схемы, функции, методы и сервисные классы должны иметь докстринги или комментарии на русском языке.
- Сложная бизнес-логика должна быть пояснена комментариями на русском языке: проверки прав, расписаний, активных смен, идемпотентности, обработки дублей и ошибок.
- Комментарии должны объяснять смысл и причину решения, а не дублировать очевидное действие кода.
- В каждом сервисе должна быть собственная документация `README.md`: назначение, переменные окружения, запуск в Docker, основные endpoint'ы, диагностика и просмотр логов.

## Главная идея

ESP32 не должен передавать доверенные `userID`, `machineID` и `workID` как главный источник истины. Сервис должен получать их через `sessionID`.

Рекомендуемый формат события:

```json
{
  "sessionID": "uuid",
  "time": "2026-05-16 20:30:00",
  "detail": {
    "number": 1,
    "type": 2,
    "state": 1
  }
}
```

Если в будущем ESP32 все же будет присылать `userID`, `machineID`, `workID`, то сервис обязан сверить их с сессией. При несовпадении событие отклонить и сохранить в `InvalidEvent`.

## Endpoint'ы

### `GET /health`

Ответ:

```json
{"status": "ok", "service": "event-service"}
```

### `POST /api/events/detail`

Назначение: принять событие создания детали.

Успешный ответ при новой детали:

```json
{"success": true, "status": "saved"}
```

Успешный ответ при дубле:

```json
{"success": true, "status": "duplicate"}
```

Ошибочный ответ:

```json
{"success": false, "error": "описание ошибки"}
```

## Проверки события

Сервис должен проверить:

```text
JSON корректно прочитан;
sessionID существует;
AuthSession активна;
AuthSession не истекла;
Work существует и status='active';
тип детали существует и активен;
состояние детали существует и активно;
detail.number положительный;
time корректно парсится в формате YYYY-mm-dd HH:mm:ss.
```

После успешной проверки:

```text
user_id берется из AuthSession;
machine_id берется из AuthSession;
work_id берется из AuthSession;
event_time берется из ESP32 time;
```

## Идемпотентность

Уникальный индекс события создания детали:

```text
UNIQUE (user_id, machine_id, work_id, detail_number)
```

При сохранении использовать:

```sql
INSERT INTO ...
ON CONFLICT (user_id, machine_id, work_id, detail_number)
DO NOTHING
```

Если запись вставлена — вернуть `saved`.
Если запись уже была — вернуть `duplicate`.

Важно: дубль не считается ошибкой для ESP32. ESP32 должен получить `success=true`, чтобы не отправлять событие бесконечно.

## InvalidEvent

Если событие невозможно обработать, сохранить его в `InvalidEvent`:

```text
received_at
source_ip
raw_body
error_text
service_name = event-service
```

Сохранять надо:

```text
битый JSON;
невалидную структуру;
несуществующий sessionID;
неактивную session;
несуществующий detail type;
несуществующее detail state;
несовпадение userID/machineID/workID, если эти поля присутствуют во входе.
```

Если событие корректно сохранено в `Detail`, raw JSON хранить не надо.

## Обновление активности смены

После успешного сохранения детали или обнаружения дубля обновить:

```text
Work.last_seen_at = now()
```

Так директор сможет видеть, что ESP32 продолжает работать.

## Репозитории и сервисный слой

Разделить код:

```text
api/routes.py                    HTTP-слой;
repositories/event_repository.py SQL-запросы;
services/event_service.py        бизнес-логика;
schemas/requests.py              Pydantic request models;
schemas/responses.py             Pydantic response models.
```

В комментариях на русском объяснить, где именно происходит идемпотентность и почему используется `ON CONFLICT`.

## Docker

`event-service` должен запускаться в контейнере командой вида:

```bash
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

Наружу напрямую порт сервиса не публиковать. Доступ через Nginx:

```text
/api/events/ -> event-service:8000
```

## Тесты

Добавить тесты:

```text
успешное сохранение детали;
повтор того же события возвращает duplicate;
битый JSON попадает в InvalidEvent;
неизвестный sessionID попадает в InvalidEvent;
неактивная session не позволяет сохранить detail;
неизвестный тип детали дает ошибку;
неизвестное состояние детали дает ошибку;
после успешного события обновляется Work.last_seen_at.
```

## Критерии приемки

- Сервис принимает событие детали.
- Сервис сохраняет detail только после проверки активной session.
- Уникальность детали обеспечивается БД.
- Дубль возвращает `success=true`, `status=duplicate`.
- Некорректные события сохраняются в `InvalidEvent`.
- В коде есть русские комментарии к бизнес-логике обработки ошибок и дублей.
