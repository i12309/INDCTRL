# Задача 04. Реализовать `auth-service` на FastAPI

## Цель

Создать FastAPI-сервис, который обслуживает ESP32 при входе работника на станок: выдает список доступных работников, проверяет пароль, создает рабочую смену, создает сессию, принимает heartbeat и завершает смену.

## Общие обязательные требования

- Проект называется `INDCTRL`.
- Язык: Python 3.12+.
- База данных: PostgreSQL.
- Запуск в production: Docker Compose на одном Linux-сервере.
- Сервисы:
  - `auth-service` — FastAPI-сервис авторизации работников на ESP32;
  - `event-service` — FastAPI-сервис приема событий о произведенных деталях;
  - `control-web` — Django-сервис, который содержит админку, dashboard и отчеты;
  - `postgres` — общая PostgreSQL-БД;
  - `nginx` — единая точка входа для HTTP.
- Админка и отчеты должны быть в одном Django-сервисе `control-web`. Внутри Django их надо разделить по приложениям/модулям.
- Django-модели и Django migrations являются основным источником структуры БД.
- FastAPI-сервисы работают с этой же БД напрямую через общий пакет и SQL-запросы, но не запускают миграции.
- Весь код должен быть понятен программисту, который плохо знает Python, FastAPI и Django.
- Все публичные классы, Django-модели, Pydantic-схемы, функции, методы и сервисные классы должны иметь докстринги или комментарии на русском языке.
- Сложная бизнес-логика должна быть пояснена комментариями на русском языке: проверки прав, расписаний, активных смен, идемпотентности, обработки дублей и ошибок.
- Комментарии должны объяснять смысл и причину решения, а не дублировать очевидное действие кода.
- В каждом сервисе должна быть собственная документация `README.md`: назначение, переменные окружения, запуск в Docker, основные endpoint'ы, диагностика и просмотр логов.

## Основная бизнес-логика

ESP32 закреплен за одним станком. Станок определяется только по MAC-адресу ESP32.

Алгоритм входа:

```text
1. ESP32 отправляет macAddress.
2. Сервис находит Device.
3. Через Device определяет Machine.
4. Работник выбирает себя из списка разрешенных работников.
5. ESP32 отправляет userID, password и macAddress.
6. Сервис проверяет пароль, права и расписание.
7. Сервис проверяет, что станок не занят другим active Work.
8. Сервис создает Work.
9. Сервис создает AuthSession.
10. Сервис возвращает sessionID, userID, machineID, workID.
```

## Endpoint'ы

### `GET /health`

Ответ:

```json
{"status": "ok", "service": "auth-service"}
```

### `POST /api/auth/device/workers`

Назначение: ESP32 получает список работников, которые могут работать на этом станке сейчас.

Запрос:

```json
{
  "macAddress": "AA:BB:CC:DD:EE:FF"
}
```

Ответ:

```json
{
  "success": true,
  "machineID": 321,
  "machineName": "Станок №1",
  "workers": [
    {"userID": 123, "fullName": "Иванов Иван Иванович"}
  ]
}
```

Фильтрация работников:

```text
пользователь активен;
роль пользователя worker;
есть разрешение UserMachinePermission.is_allowed = true;
текущее серверное время попадает в расписание UserMachineSchedule;
станок активен;
ESP32-устройство активно.
```

Если расписания нет, в первой версии считать, что работать нельзя. Это безопаснее, чем разрешать всем.

### `POST /api/auth/login`

Запрос:

```json
{
  "userID": 123,
  "password": "1234",
  "macAddress": "AA:BB:CC:DD:EE:FF"
}
```

Успешный ответ:

```json
{
  "success": true,
  "sessionID": "uuid",
  "userID": 123,
  "machineID": 321,
  "workID": 111
}
```

Проверки:

```text
macAddress существует в Device;
Device активен;
Machine активен;
User существует и активен;
пароль корректный;
роль пользователя worker;
пользователь имеет право работать на станке;
пользователь сейчас может работать по расписанию;
на станке нет другой active Work.
```

Если проверка не прошла, вернуть понятный JSON-ответ с `success=false`.

### `POST /api/auth/logout`

Запрос:

```json
{
  "sessionID": "uuid"
}
```

Действия:

```text
AuthSession.is_active = false
Work.status = finished
Work.finished_at = now()
```

Ответ:

```json
{"success": true, "status": "finished"}
```

### `POST /api/auth/heartbeat`

Запрос:

```json
{
  "sessionID": "uuid"
}
```

Действия:

```text
проверить активную сессию;
обновить Work.last_seen_at = now();
```

Ответ:

```json
{"success": true, "status": "alive"}
```

## Репозитории и сервисный слой

Разделить код:

```text
api/routes.py               только HTTP-слой;
repositories/auth_repository.py  SQL-запросы;
services/auth_service.py         бизнес-логика;
schemas/requests.py              Pydantic request models;
schemas/responses.py             Pydantic response models.
```

Не смешивать SQL-запросы и FastAPI routes в одном файле.

## Проверка занятости станка

Перед созданием смены проверить активную смену:

```text
SELECT ... FROM works WHERE machine_id = ... AND status = 'active'
```

Также полагаться на partial unique index в БД. Если два запроса придут одновременно, БД должна защитить от двойного входа на один станок.

## Работа с паролем

Пароль проверять через общий helper `verify_django_password()` из `control_common.security`.

В комментарии объяснить:

```text
Пароль не сравнивается с БД напрямую, потому что в БД хранится только hash.
```

## Docker

`auth-service` должен запускаться в контейнере командой вида:

```bash
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

Наружу напрямую порт сервиса не публиковать. Доступ к нему идет через Nginx:

```text
/api/auth/ -> auth-service:8000
```

## Тесты

Добавить тесты:

```text
получение списка работников по MAC;
ошибка при неизвестном MAC;
успешный login;
ошибка при неверном пароле;
ошибка при отсутствии разрешения на станок;
ошибка при запрете расписанием;
ошибка если станок уже занят;
logout закрывает Work и AuthSession;
heartbeat обновляет last_seen_at.
```

## Критерии приемки

- ESP32 может получить список работников по MAC.
- Успешный login создает `Work` и `AuthSession`.
- В ответе login возвращаются `sessionID`, `userID`, `machineID`, `workID`.
- `machineID` всегда определяется через MAC ESP32.
- Один станок не может получить две активные смены одновременно.
- Все ключевые функции и Pydantic-схемы имеют комментарии или докстринги на русском языке.
