# Задача 11. Настроить тесты, качество кода и базовую CI-проверку

## Цель

Добавить автоматические проверки, чтобы проект можно было безопасно развивать: unit-тесты, интеграционные тесты, форматирование, линтер и базовая GitHub Actions pipeline.

## Общие обязательные требования

- Проект называется `INDCTRL`.
- Язык: Python 3.12+.
- База данных: PostgreSQL.
- Запуск в production: Docker Compose на одном Linux-сервере.
- Сервисы:
  - `auth-service` — FastAPI-сервис авторизации работников на ESP32;
  - `event-service` — FastAPI-сервис приема событий о произведенных деталях;
  - `control-web` — Django-сервис, который содержит админку, dashboard и отчеты;
  - `postgres` — общая PostgreSQL-БД;
  - `nginx` — единая точка входа для HTTP.
- Админка и отчеты должны быть в одном Django-сервисе `control-web`. Внутри Django их надо разделить по приложениям/модулям.
- Django-модели и Django migrations являются основным источником структуры БД.
- FastAPI-сервисы работают с этой же БД напрямую через общий пакет и SQL-запросы, но не запускают миграции.
- Весь код должен быть понятен программисту, который плохо знает Python, FastAPI и Django.
- Все публичные классы, Django-модели, Pydantic-схемы, функции, методы и сервисные классы должны иметь докстринги или комментарии на русском языке.
- Сложная бизнес-логика должна быть пояснена комментариями на русском языке: проверки прав, расписаний, активных смен, идемпотентности, обработки дублей и ошибок.
- Комментарии должны объяснять смысл и причину решения, а не дублировать очевидное действие кода.
- В каждом сервисе должна быть собственная документация `README.md`: назначение, переменные окружения, запуск в Docker, основные endpoint'ы, диагностика и просмотр логов.

## Инструменты

Использовать:

```text
pytest
pytest-django
httpx / FastAPI TestClient
ruff
black
```

Опционально можно добавить `mypy`, если это не сильно усложнит проект.

## pyproject.toml

В корневом `pyproject.toml` настроить:

```text
black
ruff
pytest
```

## Makefile

Команды:

```makefile
make test
make lint
make format
make check
```

`make check` должен выполнять:

```text
lint
test
```

## Тесты common

Покрыть:

```text
парсинг времени ESP32;
проверка расписания;
проверка пароля;
ошибки конфигурации;
```


## Тестовая БД

Для тестов использовать отдельную тестовую PostgreSQL-БД.

В Docker Compose можно добавить отдельный compose-файл или profile для тестов:

```text
compose.test.yml
```

Не использовать production-БД для тестов.

## GitHub Actions

Создать:

```text
.github/workflows/ci.yml
```

CI должен:

```text
установить Python;
установить зависимости;
запустить ruff;
запустить black --check;
запустить pytest;
```

Если интеграционные тесты с PostgreSQL слишком сложны для первой версии, сделать хотя бы unit-тесты и явно описать в TODO интеграционные тесты через docker compose.

## Критерии приемки

- `make lint` выполняется без ошибок.
- `make format` форматирует код.
- `make test` запускает тесты.
- Есть CI workflow.
- Покрыты ключевые бизнес-проверки auth-service и event-service.
- В тестах нет подключения к production-БД.
