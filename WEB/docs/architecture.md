# Архитектура INDCTRL

INDCTRL переведен на Django-only архитектуру. В рабочем контуре больше нет
отдельных FastAPI-сервисов: Django отвечает за web-интерфейс, admin, dashboard,
отчеты, API для ESP32 и миграции БД.

## Общая схема

```text
ESP32   -> nginx -> indctrl -> PostgreSQL
браузер -> nginx -> indctrl -> PostgreSQL
```

Контейнеры:

- `nginx` - принимает HTTP-запросы на порту `80`;
- `indctrl` - единое приложение INDCTRL;
- `postgres` - общая PostgreSQL-БД.

## Почему Django-only

Проект стал проще: бизнес-логика авторизации, смен, деталей, dashboard и отчетов
находится рядом с Django-моделями. Не нужны отдельные FastAPI-контейнеры,
Pydantic-схемы, SQL-репозитории и дублирующая документация.

Django уже дает:

- ORM и транзакции;
- миграции;
- admin;
- авторизацию пользователей;
- формы, шаблоны и отчеты;
- обычные JSON views для ESP32 API.

## Ответственность Django

Django выполняет все прикладные функции:

- управляет пользователями, ролями, станками, устройствами и расписаниями
  допусков через admin;
- принимает `POST /api/device/workers`, `login`, `heartbeat`, `logout`, `detail`;
- создает `Work` и `AuthSession`;
- сохраняет `Detail` и `InvalidEvent`;
- показывает dashboard текущих работников;
- строит отчеты и экспорт CSV/XLSX;
- применяет миграции БД.

## PostgreSQL

PostgreSQL хранит все данные проекта. Важные бизнес-ограничения защищены на
уровне БД:

- один активный `Work` на один станок;
- уникальная деталь внутри смены по `user_id`, `machine_id`, `work_id`,
  `detail_number`;
- уникальный MAC-адрес ESP32-устройства;
- расписание как единственный источник допуска пользователя к станку.

## Nginx

Nginx проксирует все динамические запросы в `indctrl` и отдает `/static/` из
volume `static_data`. Отдельных upstream для auth/event больше нет.
